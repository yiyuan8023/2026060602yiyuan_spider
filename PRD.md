## 2. 背景

当前项目承担多个电商平台和业务系统的数据采集任务。业务侧需要持续获取店铺经营、商品、流量、交易、售后、财务、内容等数据，用于运营分析、财务核对、经营报表和内部数据沉淀。

由于各平台接口、登录态、导出格式和字段结构差异较大，项目采用脚本集合方式推进：每个平台或业务场景拥有独立的 API 封装和执行脚本，通用能力沉淀在数据库、日志、日期、Excel、Cookie 等基础模块中。

项目目前更接近“内部数据采集工具箱”，核心价值是快速支持不同平台的数据抓取与入库，而不是提供标准化的产品界面。

## 3. 项目目标

### 3.1 当前目标

- 支持多平台经营数据采集。
- 支持按店铺和日期周期采集数据。
- 支持将采集数据标准化写入 MySQL。
- 支持 Cookie 登录态维护和失效识别。
- 支持 Excel/CSV 手工文件导入数据库。
- 支持任务调度和批量执行。
- 支持运行日志记录，方便排错和追踪。

### 3.2 非目标

- 当前不提供统一 Web 管理后台。
- 当前不提供完整权限管理系统。
- 当前不保证所有平台脚本采用完全统一模板。
- 当前不承诺所有历史脚本都可直接运行，需要结合 Cookie、数据库和平台接口状态确认。
- 当前不在文档中记录真实数据库密码、Cookie、服务器地址等敏感信息。

## 6. 总体架构

```text
执行脚本层
  -> 平台 API 封装层
  -> 通用工具层
  -> 数据库存储层
  -> 日志记录层

调度器
  -> 定时触发执行脚本
  -> 并发控制
  -> 队列等待
```

### 6.1 执行脚本层

位置：

```text
执行脚本/
```

职责：

- 定义采集目标。
- 指定平台站点、店铺、表名、采集周期。
- 调用对应 API。
- 清洗字段。
- 生成唯一 key。
- 入库。

### 6.2 API 封装层

位置：

```text
API/
```

职责：

- 处理平台请求。
- 维护平台 Headers、Cookie、签名、参数。
- 处理平台特殊逻辑，例如字体解密、JS 参数生成、压缩包读取。
- 将平台返回结果交给执行脚本处理。

### 6.3 通用工具层

位置：

```text
extra/
excel/
cookie_manager/
```

职责：

- 日志。
- 数据库。
- 日期。
- 命令行参数。
- 文件下载。
- Excel/CSV 读取。
- Cookie 刷新。

### 6.4 调度层

位置：

```text
main/
```

职责：

- 定时执行采集脚本。
- 控制最大并发。
- 避免同脚本重复运行。
- 超出并发后排队执行。

## 7. 核心业务流程

### 7.1 自动采集流程

```text
维护者配置执行脚本
-> 脚本指定平台、店铺、表名、周期
-> select_shop_date 读取 Cookie 和日期
-> API 模块请求平台数据
-> 脚本清洗字段并生成 key
-> DBManager 写入 MySQL
-> logger 记录结果
```

关键规则：

- 店铺 Cookie 从数据库读取。
- 日期可使用默认最近 N 天，也可通过命令行参数覆盖。
- 目标表通常由脚本内 `table_name` 指定。
- 数据唯一性通常由脚本生成的 `key` 字段控制。

### 7.2 手工文件导入流程

```text
准备 Excel/CSV 文件
-> FileToItems 读取文件
-> pandas 解析为 DataFrame
-> 转为字典列表
-> DBManager 写入目标表
-> 日志记录导入结果
```

适用场景：

- 平台导出的账单。
- 财务对账文件。
- 售后、订单、发货等手工报表。

### 7.3 Cookie 更新流程

```text
读取旧 Cookie
-> Playwright 打开平台页面
-> 判断是否登录成功
-> 获取浏览器上下文新 Cookie
-> 写回数据库 cookie 表
```

异常场景：

- Cookie 为空。
- Cookie 已过期。
- 页面结构变化导致登录状态判断失败。
- 平台风控要求人工验证。

### 7.4 调度执行流程

```text
配置调度任务
-> 到达执行时间
-> ScriptScheduler 提交脚本
-> 检查是否正在运行
-> 未超并发则执行
-> 超并发则进入队列
-> 子进程执行 Python 脚本
-> 记录成功或失败
```

## 8. 功能需求

### 8.1 多平台 API 采集

要求：

- 支持按平台模块维护接口。
- 支持传入 Cookie。
- 支持平台特有请求参数。
- 支持接口返回字段解析。
- 支持平台异常记录。

验收标准：

- 指定店铺、日期后能成功请求数据。
- Cookie 失效时有明确日志。
- 平台字段变化时能定位到对应 API 或脚本。

### 8.2 店铺和日期选择

要求：

- 支持默认店铺列表。
- 支持采集全部店铺。
- 支持默认最近 N 天。
- 支持命令行覆盖日期范围。
- 支持日粒度和月粒度。

验收标准：

- 能从数据库读取指定站点和店铺的 Cookie。
- 能根据参数得到正确采集日期列表。

### 8.3 数据清洗和字段转换

要求：

- 支持英文键转中文字段。
- 支持补充店铺名称、统计日期等公共字段。
- 支持生成唯一 key。
- 支持平台特殊数据处理，例如字体解密、压缩文件解析。

验收标准：

- 入库字段含义清楚。
- 同一业务表的 key 稳定可复用。
- 重复采集不会产生不可控重复数据。

### 8.4 数据库存储

要求：

- 支持自动建表。
- 支持自动补字段。
- 支持批量写入。
- 支持按唯一键更新。
- 支持不同数据库配置。

验收标准：

- 空表可首次创建。
- 新字段可自动补充。
- 大批量数据可稳定写入。
- 失败时有错误日志。

### 8.5 Excel/CSV 文件处理

要求：

- 支持读取常见表格文件。
- 支持多编码 CSV。
- 支持部分加密 Excel。
- 支持文件数据入库。

验收标准：

- 文件不存在时明确报错。
- 数据读取后可转换为字典列表。
- 数据可写入指定表。

### 8.6 Cookie 管理

要求：

- 支持读取已有 Cookie。
- 支持浏览器打开并验证登录状态。
- 支持获取新 Cookie。
- 支持写回数据库。

验收标准：

- 登录有效时返回新 Cookie。
- 登录失败时记录失败原因。
- Cookie 失效日志可被识别。

### 8.7 任务调度

要求：

- 支持立即执行所有配置脚本。
- 支持固定时间执行。
- 支持 cron 表达式。
- 支持指定日期执行。
- 支持最大并发控制。
- 支持运行中脚本防重复。

验收标准：

- 任务能按配置触发。
- 同一脚本运行中不会重复启动。
- 超出并发后进入队列。

### 8.8 日志

要求：

- 支持普通日志和错误日志。
- 支持按日期轮转。
- 支持文件日志开关。
- 支持 Cookie 相关错误收集。

验收标准：

- 每次执行有开始、结束、失败信息。
- 错误日志可快速定位脚本和行号。

## 9. 权限与安全

### 9.1 平台访问权限

项目依赖各平台登录 Cookie 获取接口数据。Cookie 代表平台访问身份，必须按敏感凭证处理。

要求：

- 不在文档中记录真实 Cookie。
- 不在提交信息中记录真实 Cookie。
- Cookie 表访问权限应限制在维护者和服务账号。
- Cookie 失效后应能快速定位对应店铺和平台。

### 9.2 数据库权限

项目依赖 MySQL 存储采集数据和 Cookie。

要求：

- 数据库地址、用户名、密码应通过环境变量或配置项管理。
- 文档中只写占位符，不写真实密码。
- 采集账号原则上只授予需要的库表权限。
- 高风险操作，如删除后插入，应先小范围测试。

推荐环境变量：

```text
MYSQL_HOST=<数据库地址>
MYSQL_PORT=<数据库端口>
MYSQL_DB=<数据库名或环境名>
MYSQL_USER=<数据库用户名>
MYSQL_PASSWORD=<数据库密码>
LOGFILE=<是否写文件日志>
PYTHONIOENCODING=utf-8
```

### 9.3 本地文件权限

项目会读取本地 Excel/CSV 文件，也会生成日志和打包产物。

要求：

- 财务文件、账单文件不要随意提交到 Git。
- 日志中可能出现店铺名、表名、接口错误信息，应谨慎外发。
- 打包产物和缓存文件应与源码分离。

## 10. 数据设计

### 10.1 数据来源

| 来源 | 类型 |
|---|---|
| 平台接口 | JSON、压缩包、Excel 下载链接 |
| 本地文件 | Excel、CSV、TXT |
| Cookie 表 | 店铺登录态 |
| 命令行参数 | 日期、店铺等运行参数 |

### 10.2 数据去向

主要去向为 MySQL 业务表。

典型表命名方式：

```text
平台_模块_页面_指标_版本或月份
```

示例类型：

- 商品明细表。
- 流量来源表。
- 交易概况表。
- 售后服务表。
- 财务账单表。
- 手工导入表。

### 10.3 主键和唯一键

多数脚本会生成 `key` 字段作为业务唯一键。常见组成：

```text
店铺名称 + 商品ID/SKU/订单号 + 统计日期
```

维护要求：

- 新脚本必须明确唯一键规则。
- 同一业务表不应频繁改变 key 组成。
- 更新历史数据前应确认 key 是否覆盖正确。

### 10.4 字段管理

项目支持根据数据自动建表和自动补字段。

优点：

- 新接口上线快。
- 平台字段变化时不一定需要手工改表。

风险：

- 字段命名不统一会造成表结构膨胀。
- 中文字段变化会影响下游使用。
- 自动补字段不能替代字段口径管理。

## 12. 当前主要问题

### 12.1 仓库产物污染

现象：

- 存在大量 `__pycache__`。
- 存在历史日志。
- 存在打包产物、`.exe`、`build` 文件。

影响：

- Git 状态噪声大。
- 不利于区分真实代码变更。
- 仓库体积容易膨胀。

### 12.2 敏感配置暴露风险

现象：

- 配置文件中存在数据库连接相关配置。
- Cookie 和账号相关信息依赖数据库或脚本运行环境。

影响：

- 账号和数据安全风险。
- 不利于多环境部署。

### 12.3 编码和中文路径风险

现象：

- 文件名大量使用中文。
- 部分终端或 Git 输出中可能出现中文乱码。
- 调度配置中的脚本路径若编码异常，会导致脚本找不到。

影响：

- 跨环境迁移成本高。
- 排查路径问题费时。

### 12.4 脚本规范不统一

现象：

- 多数脚本独立维护表名、店铺、周期、字段映射。
- 公共流程重复较多。

影响：

- 新增脚本快，但长期维护成本增加。
- 容易出现主键规则、字段命名、错误处理不一致。

### 12.5 平台接口变动风险

现象：

- 平台接口、签名、风控、页面结构可能变化。
- Cookie 登录态可能失效。

影响：

- 采集任务突然失败。
- 需要快速定位是 Cookie、接口、字段还是风控问题。

## 13. 迭代计划

### 13.1 第一阶段：仓库治理

目标：

- 降低维护噪声。
- 保护敏感信息。
- 提升接手效率。

事项：

- 补充 README 和 PRD。
- 建立 `.gitignore`，忽略日志、缓存、打包产物。
- 清理不应进入版本管理的历史产物。
- 标记必须保留的 `.exe` 或业务交付文件。
- 将敏感配置迁移为环境变量或 `.env`。

验收标准：

- `git status` 能清晰展示真实源码变更。
- 文档能指导维护者定位脚本和模块。
- 文档不出现真实敏感凭证。

### 13.2 第二阶段：脚本模板统一

目标：

- 降低新增采集任务成本。
- 减少重复代码。

事项：

- 总结标准采集脚本模板。
- 统一店铺、日期、表名、key、入库、日志流程。
- 为新脚本定义必填元信息。
- 梳理历史脚本，优先改造高频任务。

验收标准：

- 新增同平台脚本可按模板快速落地。
- 常见错误处理方式一致。

### 13.3 第三阶段：调度配置外置

目标：

- 避免调度脚本路径写死。
- 降低中文路径和编码风险。

事项：

- 将任务配置从 Python 文件迁移到 JSON/YAML。
- 为每个任务定义名称、脚本路径、调度时间、并发策略、启停状态。
- 启动前校验脚本路径是否存在。
- 输出调度任务清单。

验收标准：

- 不改代码即可增删调度任务。
- 路径错误能在启动时被发现。

### 13.4 第四阶段：运行观测增强

目标：

- 更快发现失败任务。
- 更快定位失败原因。

事项：

- 标准化每个脚本的开始、结束、耗时、数据量日志。
- 区分 Cookie 失效、接口失败、数据为空、入库失败。
- 增加任务结果汇总。
- 可选接入邮件、钉钉或其他通知。

验收标准：

- 每次任务能看出是否成功、采集多少、失败原因。
- Cookie 失效能明确到平台和店铺。

### 13.5 第五阶段：数据口径治理

目标：

- 降低下游用数风险。
- 提升字段稳定性。

事项：

- 梳理核心表清单。
- 记录每张表的数据来源、主键、粒度、更新频率。
- 统一字段命名规范。
- 对核心表建立最小数据校验。

验收标准：

- 核心业务表有口径说明。
- 字段变化能被记录和追踪。

## 14. 验收标准

项目当前维护验收标准：

- 能明确找到某个业务脚本的位置。
- 能明确脚本依赖哪个 API 模块。
- 能明确脚本写入哪个表。
- 能明确 Cookie 从哪里来。
- 能明确采集日期如何生成。
- 能明确错误日志在哪里看。
- 新增脚本时不泄露敏感配置。
- 调整配置时不误提交日志、缓存和打包产物。

## 15. 维护检查清单

每次新增或调整采集脚本前，检查：

- 是否确认平台和业务页面。
- 是否确认目标表名。
- 是否确认唯一 key。
- 是否确认默认店铺。
- 是否确认默认采集周期。
- 是否确认 Cookie 站点名称。
- 是否小范围日期测试。
- 是否检查日志。
- 是否确认无敏感信息进入文档或提交。

每次排查失败任务时，检查：

- 脚本路径是否正确。
- Cookie 是否失效。
- 平台接口是否变动。
- 返回数据是否为空。
- 字段映射是否变化。
- 数据库连接是否正常。
- 目标表是否存在结构异常。
- 日志是否有明确错误。

## 16. 结论

`yiyuan_spider` 当前是一个实用优先的内部数据采集项目，已经具备多平台采集、文件导入、Cookie 维护、数据库写入和任务调度能力。

短期重点不是重写，而是先治理仓库、保护配置、补齐文档、统一脚本模板。等这些基础打稳，再考虑调度配置外置、运行观测和数据口径治理。
