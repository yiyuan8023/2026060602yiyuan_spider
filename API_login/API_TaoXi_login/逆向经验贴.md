# 淘系 Havana 登录逆向实战

> 本文覆盖淘系 Havana 登录体系下两个站点的逆向全过程：
> - **Part A**：生意参谋（SYCM） — 从"混合方案"到纯协议无浏览器
> - **Part B**：DChain 供应链 — 从页面分析到纯协议 + iframe 自动化兜底

> 逆向通用经验见文末第 9 节。

## 0. TL;DR（结论先行）

- **RGV587 风控的真因不是滑块，是 `umidToken` / `bx-ua` 缺失或太弱**。补齐合格令牌后，密码登录直接放行，**登录页根本不出滑块**。
- **um.js / collina.js 这类 JSVMP，"无法脱浏览器"是旧结论**。换 **iv8（C++ 原生浏览器 API 的真 V8）**，JSVMP 照跑，令牌纯协议拿到，全程零浏览器、~4s。
- **真正的拦路是登录后的 IV 二次身份验证**（`passport.taobao.com/iv/...`），实测默认下发**短信**而非滑块，可纯协议求解。
- **令牌可重放**：`um.json` 的指纹载荷在纯 Python 里重放仍能换出有效 `umidToken` —— 纯算局部可行，但全量纯算（手写复现 JSVMP）不划算，iv8 是甜点位。

最终链路：

```
iv8 生成 umidToken+bx-ua(无浏览器, ~4s)
  → requests POST login.do（破 RGV587）
  → 命中 IV 则纯协议短信求解（send_code → 邮箱读码 → 提交 J_Form）
  → 校验登录态有效 → 写 get_cookie
失败再降级：浏览器自动化兜底（已修复 IV 验证页处理与"假成功"）
```

## 1. 逆向目标

把 `https://login.taobao.com/havanaone/loginLegacy/password/login.do` 的卖家后台登录做成**无人值守、无可见浏览器、协议级**，产出可直接用于生意参谋采集的会话 Cookie（`unb` / `cookie2` / `sgcookie` / `_tb_token_` 等）。

接手时的现状（旧方案）：headless DrissionPage 提取 `umidToken`/`bx-ua` → 协议登录。问题描述是"**协议登录遇到风控，自动化登录后没验证最后的滑块就关掉了**"。

这句话本身就埋了一个**误判**，见下一节。

## 2. 最初的难点与误判（这是全场最值钱的部分）

### 误判一："卡在滑块" —— 其实滑块只是症状

**问题**：用户认为登录被"滑块"卡住。

**分析**：先不信结论，直接看协议响应。带旧 token POST `login.do`，拿到的是 `RGV587_ERROR`，而不是任何滑块挑战。`RGV587` 是阿里风控的"设备不可信"统一错误码。

**证据**：旧 `extract_security_tokens()` 用 headless 提取，SKILL_SUMMARY 自己都记过"headless 模式 umid 为空"——也就是**令牌时有时无、质量不稳**。

**结论**：`RGV587` 的病根是**安全令牌缺失/太弱**，不是滑块。滑块是风控升级后才出现的二级现象。**先定位错误码，再谈绕过**，否则会一头扎进"过滑块"的死胡同。

### 误判二："JSVMP 无法脱浏览器"

**问题**：旧总结判定 `um.js` 是 JSVMP + 依赖 Canvas/WebGL，"无法在 Node.js 运行"，于是只能开浏览器取 token。

**分析**：Node/jsdom 跑不动，是因为补不齐 Canvas/WebGL/字体等指纹 API。但 **iv8 在 C++ 层原生实现了 200+ 浏览器 API**，本质是一个带浏览器环境的真 V8——JSVMP 是 JS，在真 V8 里当然能跑，缺的环境 iv8 已经补上。**旧结论的前提（"补不齐环境"）在 iv8 上不成立。**

**结论**：换工具，假设就翻盘。这条是整轮突破的支点。

### 误判三：浏览器兜底的"假成功"污染数据库

**问题**：自动化登录"没验证滑块就关了"。

**分析**：兜底判定登录成功的逻辑是 `"login" not in page.url`。而 IV 验证页是 `passport.taobao.com/iv/normal_validate.htm`——**URL 里不含 "login"**，于是被误判为"已离开登录页 = 成功"，把验证页的无效 Cookie 当成功写进了 `get_cookie`。

**结论**：这才是"没过验证就关掉"的真身——不是没处理验证，是**把验证页当成了已登录**。判定登录态要白名单（确实到了卖家后台），不能用"不含 login"这种黑名单。

### 误判四："IV 是滑块" —— 实测默认是短信

见第 5 节。抓下来才知道服务器默认下发的是短信验证页，不是滑块。

## 3. 分析路径（怎么一步步走到纯协议）

整条路径的节奏是：**先证伪最危险的假设，再顺藤摸瓜**。

### 3.1 命门优先：iv8 到底能不能跑 um.js

不写任何业务代码，先一发冒烟脚本：`page.load` 一个空页 → 补 `MessageChannel` → 顺序 `eval` `awsc_baxia.js` + `um.js` + `collina.js` → 探全局。

先搞清楚 AWSC（百夏/baxia）的模块地图，知道每个 `AWSC.use(...)` 拉的是什么、产出什么：

```
baxiaCommon.js / awsc_baxia.js (编排器)
├── AWSC.use("um")   → um.js (178KB, JSVMP)     → umidToken (设备指纹) → POST ynuf um.json
├── AWSC.use("uab")  → collina.js (249KB, JSVMP)→ bx-ua (行为指纹, 140#...) → fy.getUA()
├── AWSC.use("et")   → et_f.js                  → 事件追踪 (um 流水线会先拉它)
├── AWSC.use("fy")   → fireyejs.js              → 风控引擎
└── AWSC.use("nc")   → nc.js                    → 滑块验证码 (本轮未用到——令牌合格后不出滑块)
```

CDN base 是 `https://g.alicdn.com/AWSC/...`。我们只需要 um + uab 两条线（umidToken + bx-ua），nc 这条线本轮根本没碰到。

**证据**（冒烟输出，关键行）：

```
typeof window.AWSC => object
AWSC.use('um') 回调 => init_cb:success
netLog: GET et_f.js → GET ynuf.aliapp.org/w/wu.json → POST ynuf.aliapp.org/service/um.json
```

`AWSC` / `baxiaCommon` / `um` 全部就位，um 模块 `init:success`，netLog 抓到它真实想发的请求链。**命门通了**——剩下都是工程问题。

### 3.2 复现 SDK 的网络流水线

iv8 社区版不发真实网络，请求只进 `netLog`。流水线模式：

```
推进事件循环(eventLoop.sleep) → 读 netLog 新请求 → requests 真实发出 → add_resource 注入响应 → 再推进
```

um 的请求是**迭代式**的（et_f.js 脚本 → wu.json → 最后 POST um.json），所以注入必须做成循环，逐轮喂。

**坑（见 7.1）**：`add_resource` 一上来报"资源 bundle 尚未初始化"——必须先 `__iv8__.page.load(...)` 而不是 `innerHTML`。

### 3.3 umidToken 怎么拿——别跟 SDK 的异步回调死磕

um 的 `init` 回调会**乐观返回 `success` 但 `data.tn` 是空的**（真 token 在异步 POST 回来之后）。与其追回调时序，不如**直接解析 `um.json` POST 的响应**：

**证据**：

```
um.json 响应 keys: ['tn','id']
tn = T2gADrP1iGPf28yn...  (68 位)
```

`tn` 就是 umidToken。指纹生成（JSVMP 那段）在 iv8 里跑，token 在 Python 侧解析响应拿到。**把"难的"留给真 V8，把"取值"留在 Python**。

### 3.4 bx-ua 怎么拿——configFY 的签名是个坑

`baxiaCommon.getUA()` 一开始返回哨兵 `default_not_function`，因为 uab 模块没初始化。读 `awsc_baxia.js` 发现 getUA 被 `AWSC.configFY` gate 住。

**坑（见 7.2）**：`AWSC.configFY({...}, cb)` 直接报 `t is not a function`。回看源码 `configFY=function(e,n,o,a){return t(e,n,o,a)}`，内部 `f` 里有 `t(i)`——**第一个参数是回调函数，不是 config**。

正确姿势：

```js
AWSC.configFY(function(fy){ window._fy = fy; }, {appName:'taobao', bizName:'taobao'});
```

**证据**：跑完注入循环后 `window._fy.getUA()` 返回 `140#FHfxpv...`（672 位，`140#` 是 bx-ua 标准头）。

### 3.5 真实登录——验证假设

把 iv8 令牌接进协议 `login()`，真跑一次：

**证据**：

```
umidToken:(有) ua:(有)
login.do => {"content":{"data":{"loginResult":"success","redirectUrl":"https://mai.taobao.com/seller_admin.htm"...}}}
status = success, 21 cookies (unb/cookie2/sgcookie/_tb_token_)
```

**林内官方旗舰店直接登录成功，零滑块**。RGV587 被合格令牌正面破掉。误判一坐实。

### 3.6 发现 IV，并抓清楚它的链路

跑全量 4 店时，3 店返回的不是 success：

```
{"data":{"loginResult":"","redirectUrl":"https://passport.taobao.com/iv/remote/pc/login_check.htm?havana_iv_token=...","resultCode":100}}
```

注意：**`resultCode:100` 不是判别位**（成功的店也是 100）。判别是 **`loginResult==""` 且 redirect 落到 `/iv/`**。

顺着 redirect 抓链路（纯 requests + bs4）：

```
login_check.htm → normal_validate.htm(pointman 引导, 采 _iv_umidfg) → verify_modes.htm → identity_verify.htm?tag=8
```

`identity_verify.htm` 里有"获取短信校验码"按钮 ×5——**这一关默认是短信，不是滑块**（误判四）。

## 5. 关键接口 / 参数 / 签名 / 加密点定位

### 5.1 登录接口

- `GET https://login.taobao.com/havanaone/login/login.htm?bizName=taobao`
  从 HTML 里 `window.viewConfig` / `window.viewData` 取：
  - `rsaModulus` / `rsaExponent`（`10001`）—— 密码 RSA 公钥
  - `api.loginApi` = `/havanaone/loginLegacy/password/login.do?...`
  - `loginFormData._csrf`
- `POST {loginApi}`，关键字段：
  - `loginId`、`password2`（RSA PKCS1v1.5 加密后 hex）
  - **`umidToken`**、**`ua`（bx-ua）**、`umidTag=WEB`
  - `_csrf` + loginFormData 透传

### 5.2 两个安全令牌的来源与格式

| 令牌 | 来源 | 格式/定位 |
|---|---|---|
| `umidToken` | `POST https://ynuf.aliapp.org/service/um.json`，载荷 `data=107!<加密指纹blob>`（~551B），响应 `{"tn","id"}` | 取响应 `tn`，68 位，`T2gA...` 开头 |
| `bx-ua` | collina(uab) 模块，`AWSC.configFY(cb, cfg)` 初始化后 `fy.getUA()` | `140#...`，~670 位 |

加密点都在 `107!` / `140#` 这两个 blob 内部（JSVMP 产出，明文里看不到时间戳），**不要试图静态解开**。

### 5.3 IV 二次验证（短信模式）

- 判别：`loginResult==""` 且 redirect 含 `/iv/`（不要用 `resultCode`）
- 链路：`login_check.htm → normal_validate.htm → verify_modes.htm → identity_verify.htm`
  - `normal_validate` 是 pointman 指纹引导页，URL 里 `_umidfg` 直接置 `1` 即可达短信表单（pointman 非强制）
- 发码：`GET https://passport.taobao.com/iv/phone/send_code.do?htoken=...`（带 `phone`(掩码)/`area`/`tag`）
- 提交：`POST` 回 `identity_verify.htm`，表单 `#J_Form` 字段：
  - `_fm.v._0.ph` = 短信验证码（`#J_Phone_Checkcode`）
  - `_fm.v._0.h` = htoken、`_fm.v._0.p` = 掩码手机、`_fm.v._0.a` = 区号、`_tb_token_`、`action=verify_action`
- 成功标志：跳 `havanaone/login/continue.htm`，会话回到登录态

baxia 通过 `baxiaCommon.init({checkApiPath: url=>/iv/...})` 拦截所有 `/iv/` 请求注入 x5sec 签名——**但实测短信路径在协议层直接走通，未强制 x5sec**，省了一大块。

## 6. 最终怎么还原 / 绕过

### 6.1 令牌（纯协议，iv8）

配方（已固化进 `protocol_login/iv8_token_provider.py`）：

1. `page.load` 空页（**必须**，初始化资源 bundle）+ `wrapNative` 补 `MessageChannel`
2. `eval` `awsc_baxia.js` + `um.js` + `collina.js`
3. `AWSC.configFY(cb, {appName:'taobao',bizName:'taobao'})`（**回调在前**）
4. 注入循环：`netLog → requests → add_resource → eventLoop.sleep`
5. `umidToken` = 解析 `um.json` 响应 `tn`；`bx-ua` = `fy.getUA()`
6. 顺带把流水线里 requests 攒到的真实淘宝 Cookie（cna 等）带回登录会话

失败回落旧的 DrissionPage 提取，保证不至于全挂。

### 6.2 IV 短信（纯协议，`protocol_login/iv_sms_solver.py`）

跟链路 → 解析表单 → `send_code.do` 发码 → IMAP 邮箱读码 → POST `J_Form` → 跳 `continue.htm`。

### 6.3 Cookie 策略（三需求落地）

- **有效则刷新回存**：DB Cookie 有效时，用它访问卖家后台，捕获服务端 `Set-Cookie` 滚动刷新（滑动续期 + 轮转令牌 `isg`/`tfstk`/`_m_h5_tk` 更新），覆盖写回 `get_cookie`。不重登、不发短信。
- **失效走纯协议**（6.1 + 6.2）
- **协议失败走浏览器自动化兜底**，并修复其 IV 验证页处理（`#J_GetCode`/`#J_Phone_Checkcode`/`#submitBtn`）与"假成功"误判
- **写库前再校验**新 Cookie 有效，校验通过才覆盖，绝不用无效 Cookie 盖库

### 6.4 纯算可行性（做了实验，结论是"不划算"）

抓一份 `um.json` 载荷，**纯 Python 重放** → HTTP 200 返回**有效新 token**，且能真登录。说明令牌端**可重放**、服务端按载荷签发。

但：① `107!`/`140#` 是 JSVMP 产物，从零复现是数周/单令牌的工程且版本一升即废；② bx-ua 含时效，静态复用有 TTL + 风控标记风险。**iv8 已拿到纯算 95% 的好处（无浏览器/纯协议/快），还自适应版本升级——这才是甜点位。**

## 7. 踩坑记录（按"会卡你多久"排序）

1. **`add_resource` 报"资源 bundle 尚未初始化"** —— 必须先 `__iv8__.page.load({...})`，不能用 `innerHTML` 建 DOM。bundle 是 page.load 才建的。
2. **`configFY` 报 `t is not a function`** —— 第一个参数是 **callback(fy)**，不是 config 对象。源码 `t(i)` 出卖了它。
3. **um `init` 回调 success 但 `tn` 为空** —— 回调是乐观返回，真 token 在异步 POST 响应里。**直接解析 `um.json` 响应的 `tn`**，别等回调。
4. **`getUA()` 返回 `default_not_function`** —— uab 没初始化的哨兵值；必须 `configFY` 跑通后用 `fy.getUA()`。
5. **IV 提交落到 `iv/static/expired.htm`** —— 求解器**重复遍历了一次 IV 链路**，刷新了 htoken，导致发码用的 htoken 和提交用的对不上。**只能遍历一次**，发码与提交共用同一 htoken / 同一表单页 URL。
6. **浏览器兜底"假成功"** —— `iv/normal_validate.htm` 不含 "login" 被判成功，写入无效 Cookie。判定登录态要白名单（到卖家后台），并显式排除 `/iv/`、`normal_validate`、`login_check`、`punish`。
7. **`resultCode:100` 不是 IV 判别位** —— 成功和 IV 都是 100。判别用 `loginResult==""` + redirect 含 `/iv/`。
8. **iv8 脚本里不能用 `Date.now()`/`Math.random()`/无参 `new Date()`** —— 会破坏 resume；时间戳从外部传入。
9. **gitignore `_*.py` 误伤 `__init__.py`** —— `*` 把 `_init_` 也匹配了。改 `_[!_]*.py`，否则新建的包 `__init__.py` 被悄悄忽略、漏提交。
10. **Cookie 跨子域** —— 协议会话注入浏览器 Cookie 时显式 `domain=".taobao.com"`，否则 `myseller` 子域用不上。

### 历史方案为何废弃（别再走回头路）

下面这些在旧 `逆向过程总结.md` / `SKILL_SUMMARY.md`（2026-06-14/15）里曾是"成果"或"待办"，现在都已被取代或证伪。**列在这里就是为了下次不再重蹈：**

11. **"headless 浏览器取 token + 协议登录"混合方案（旧主路径，~14s）已被纯协议取代。** 失效原因：headless 提取 `umidToken` 时有时无（不稳）、起浏览器慢（~8s）、依赖 Chrome 进程。iv8 直接跑 JSVMP，~4s 无浏览器。**别再默认"安全 token 只能从浏览器取"。**

15. **旧"待完成：iv8 验证码求解集成到主流程"作废。** 失效原因：服务器默认走短信、登录页和 IV 都不出滑块，集成滑块求解没有落点。

# Part B: DChain 供应链 — 从页面分析到纯协议 + iframe 自动化兜底

> 账号上下文：`bc_yiyuan`，站点 DChain 供应链 (`web.scm.tmall.com`)。
> DChain 和 SYCM 共用 Havana 登录系统（RSA + iv8 + AWSC），差异在入口参数和 session 建立方式。

## 1. 结论先行

- **DChain 登录页不是直接跳转 Havana，而是 iframe 嵌入**。页面有 4 个 iframe，"普通登录" Tab 对应 `iframe#alibaba-login-box` → `havanalogin.taobao.com/mini_login.htm`。
- **纯协议登录完全走通**：iv8 令牌 → POST `/newlogin/login.do?appName=ascp&fromSite=31` → 返回 st/st2/st3/asyncUrls，全程 ~2 秒。
- **登录成功后需要 3 步 ST 回调**：basic_login_proxy → updateSession → autoLogin，建立 DChain 域 session。
- **浏览器自动化兜底有 iframe 坑**：表单和滑块都在 iframe 内，DrissionPage 必须 `get_frame()` 进入 iframe 上下文才能操作。

最终链路：

```
iv8 生成 umidToken+bx-ua (无浏览器, ~2s)
  → GET havanalogin.taobao.com/mini_login.htm?appName=ascp (提取 RSA/loginApi/loginFormData)
  → POST /newlogin/login.do?appName=ascp&fromSite=31
  → 跟 asyncUrls (补全跨域 Cookie)
  → 3步 ST 回调 (basic_login_proxy → updateSession → autoLogin)
  → 校验登录态有效 → 写 get_cookie
风控/IV 则：纯协议短信求解 或 浏览器自动化兜底（需进 iframe）
```

## 3. 协议登录差异

### 3.1 POST 登录请求

```
POST https://login.taobao.com/newlogin/login.do?appName=ascp&fromSite=31
```

和淘系格式相同但多了 ascp 特有字段：
- `appName`: ascp（淘系是 taobao）
- `appEntrance`: ascp_default
- `returnUrl`: https://web.scm.tmall.com/login
- `fromSite`: 31

### 3.2 登录成功响应格式差异

```json
{
  "content": {
    "data": {
      "st2": "tb_s_ascp_1...",
      "st": "tb_s_ascp_1...",
      "st3": "tb_s_ascp_1...",
      "asyncUrls": ["https://login.taobao.com/jump?target=...", "..."]
    }
  }
}
```

**和淘系的关键区别**：
- 没有 `ret` 字段（淘系有 `"ret": ["SUCCESS"]`）
- 没有 `loginResult` 字段
- 没有 `redirectUrl` 字段
- 成功标志是 **`st` + `asyncUrls` 同时存在**

### 3.3 ST 回调建立 DChain Session（4 步）

拿到 st 后不能简单访问 `returnUrl?st=xxx`，需要走 4 步：

1. **basic_login_proxy**：`GET web.scm.tmall.com/pages/ascm/basic_login_proxy?action=login&st=xxx&st2=xxx`
2. **tenants 查询**：`GET permission-o.scm.tmall.com/views/tenants?_tag=default` → 获取 `merchantCode`（_mcode1_）、`shortCode`（_mcode2_）、`merchantType`
3. **updateSession**：`POST session.scm.tmall.com/session/updateSession`（Set-Cookie: X-XSRF-TOKEN）
4. **autoLogin**：`GET scm.tmall.com/basic/autoLogin?_mcode1_=supplier_xxx&_mcode2_=TMYPDQ&locale=zh-cn&merchantType=100011`（Set-Cookie: _nk_）

autoLogin 的商家参数必须从 tenants 响应动态获取。缺少这些参数会导致 session 建立不完整（后续 Cookie 验证时被重定向回登录页 → `new_cookie_invalid`）。

## 5. 踩坑记录

1. **误判"DChain 直接跳转 Havana"。** 页面 URL 没有跳转，表单在 iframe 内。浏览器自动化必须先 `get_frame(iframe)` 进入 iframe 才能操作元素。先看 iframe 列表再下结论。

2. **滑块 NoRectError（iframe截图限制）。** DrissionPage 在 iframe 上下文中找到的元素虽然 `is_displayed=True`，但调用 `get_screenshot()` 会报 `NoRectError: 该元素没有位置及大小`。这是DrissionPage处理iframe内元素截图的已知限制。**错误做法**：在iframe上下文（`ctx = page.get_frame(...)`）中对滑块元素调用 `get_screenshot()`。**正确做法**：滑块处理函数传入主页面对象 `page` 而不是 iframe上下文 `ctx`，让DrissionPage在主页面上下文自动穿透iframe定位元素。修复位置：`browser_fallback.py:236` 改为 `handle_nc_slider(page, ...)` 而不是 `handle_nc_slider(ctx, ...)`。

3. **协议登录"未识别的响应结构"。** DChain newlogin 响应没有 `ret`/`loginResult`，旧代码判定为 unknown。修复：增加 `data.get("st") and data.get("asyncUrls")` 成功判定条件。

4. **Cookie 验证要求 `unb`。** 旧代码强制要求有 `unb`（淘宝用户ID），DChain 登录可能没有。修复：放宽为只要求 `cookie2` 存在。

5. **登录成功但 DChain 后台拒绝。** asyncUrls 只是跨域 Cookie 补全，没有在 DChain 服务端建立 session。修复：登录成功后走 3 步 ST 回调（通过 `st_callback` 参数传入站点特定逻辑）。

6. **操作过频触发风控。** 短时间多次登录返回 `RGV587_ERROR::SM::操作过频`。教训：调试注意间隔；正式部署 Cookie 刷新优先不会频繁登录。

7. **POST 域名必须和 GET 登录页一致。** `loginApi` 是相对路径（如 `/newlogin/login.do?...`），必须拼接到 GET 页面的实际域名（`havanalogin.taobao.com`），不能硬编码 `login.taobao.com`。域名不对时 CSRF 校验失败，服务端返回"请输入密码"（`resultCode: 100`）——表面是密码问题，实质是 session 不匹配。修复：从 GET 响应的 `r1.url` 解析 `scheme://netloc` 作为 POST base。

8. **iv8 cookies 会污染 CSRF。** iv8 生成的副产品 cookies（`cookie2`、`_tb_token_` 等）设到 `.taobao.com` 域后，和 GET `havanalogin.taobao.com` 返回的同名 cookies 冲突——requests 的 cookie jar 同时保留两组，导致 POST 时 `_csrf_token` 与 session cookie 不匹配。修复：不预设 iv8 cookies 到 login session，只取 `umidToken` 和 `ua`。

9. **`/newlogin/login.do` 密码字段同时发 `password` 和 `password2`。** SYCM 旧接口 (`/havanaone/loginLegacy/`) 只认 `password2`，DChain 新接口 (`/newlogin/`) 只认 `password`。为兼容两者，同时发送两个字段（多余字段被服务端忽略）。只发 `password2` 到 newlogin 会得到 `resultCode: 100`。

10. **操作过频的惩罚性响应伪装成"请输入密码"。** 频率限制触发后，服务端不一定返回 `RGV587_ERROR`，有时直接返回 `resultCode: 100, titleMsg: "请输入密码"`。这不是密码/字段名问题，是反爬的混淆手段。判断方式：同样的代码同样的请求，间隔一段时间后能正常返回 `RGV587`/`SUCCESS`。

## 9. 下次遇到类似站点的通用经验

1. **先定位错误码，再谈绕过。** 用户说"卡在滑块"十有八九是症状。先看协议响应：是 `RGV587`(设备不可信) / `FAIL_SYS_USER_VALIDATE`(账密) / 还是真挑战。病根错了，后面全白做。

2. **风控的本质是设备身份。** 阿里系（淘宝/1688/AliExpress 都是 baxia/AWSC）的登录拦截基本都是 `umidToken`(设备) + `bx-ua`(行为) 这两类指纹。先把这俩补齐，很多"滑块"压根不会出现。

3. **JSVMP 不要硬逆，用真 V8 执行。** Node/jsdom 补不动的环境，iv8（或等价的 C++ 引擎级补环境）能跑。**让真 V8 算 JSVMP，你在 Python 侧取值**。额外红利：它执行真实 JS，站点升级时自动适配，比手写纯算耐操得多。

4. **复现 SDK 网络用"netLog → 真发包 → 注入 → 推进"四步循环。** 离线引擎不发网，但能记录 JS 想发什么；你在外面替它发、把响应塞回去、推进事件循环。指纹换 token 这类"提交载荷拿结果"的接口，这套通杀。

5. **能取值就别追时序。** SDK 的异步回调时序很坑（乐观 success、空字段）。如果目标值最终来自某个网络响应，**直接解析那个响应**，比追回调稳得多。

6. **令牌往往可重放。** 很多"指纹换 token"接口是服务端按载荷签发、非单次失效。抓一份载荷纯 Python 重放试一发，能判断"必须现场生成"还是"可缓存复用"，直接决定要不要上重型方案。

7. **二次验证要抓全链路、认清模式。** `loginResult` 空 + 跳 `/iv/` 是入口，但具体是短信/滑块/扫码要跟到最后一跳（这里是 `identity_verify.htm`）才知道。模式不同，求解完全不同——别假设是滑块。

8. **多级降级保可用。** 协议(主) → 协议级验证求解(短信) → 浏览器自动化(兜底) → 人工介入(最后)。任意一级挂了还有下一级，生产环境比"一条路走到黑"稳得多。

9. **判定"成功"要白名单，写入要先校验。** "URL 不含 login = 成功"这种黑名单判定会被验证页骗。到没到目标态，用正向特征确认；写库前再 validate 一次，别拿没验证的状态污染数据。

10. **每改完先静态自检。** `py_compile` + 导入探针 + 单元小跑，把 80% 的低级错误挡在真请求之前——逆向阶段真请求很贵（会发短信、会触发风控频率限制）。
