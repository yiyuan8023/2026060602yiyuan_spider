# 淘宝子账号（DChain）纯协议登录 — 抓包分析与现状

> 目标账号形如 `主账号:子账号`（如 `林内供应商:BI`），通过 DChain「淘宝登录」tab 登录。
> 本文记录真实登录流量的协议还原，以及纯协议复刻的进展与卡点。

## 一、完整协议流程（浏览器抓包还原）

登录表单在 **`login.taobao.com/member/login.jhtml?...sub=true&from=ascp`** 的跨域 iframe 内
（注意：不是 `havanalogin.taobao.com/mini_login.htm`）。

```
1. GET  login.taobao.com/member/login.jhtml?style=miniall&sub=true&sub_jump=current
        &css_style=ascp_cs_tb&from=ascp&newMini2=true&keyLogin=true
        &tpl_redirect_url=<havanalogin.taobao.com/trustLogin/taobaoTrustLogin.htm?appName=ascp
                           &returnUrl=web.scm.tmall.com/pages/ascm/basic_login_proxy?action=login>
        → 页面内联 viewConfig/viewData（结构同 mini_login），rsaModulus = d3bcef1f...（同一把 key）

2. POST login.taobao.com/newlogin/account/check.do?appName=taobao&fromSite=0
        body: loginId=<主账号:子账号> & ua=140#...（bx-ua）
        resp: {"content":{"data":{"resultCode":100,"processFinished":true},"success":true}}

3. POST login.taobao.com/newlogin/login.do?appName=taobao&fromSite=0
        body: loginId=<主账号:子账号> & password2=<RSA密文 HEX, 256字符> & keepLogin=false
              & isIframe=true & documentReferer=web.scm.tmall.com/login & defaultView=password & ua=140#...
        resp(成功): {"content":{"data":{"resultCode":100,"iframeRedirect":true,
                     "iframeRedirectUrl":"login.taobao.com/member/login_unusual.htm?user_en_id=...
                       &tpl_redirect_url=...&returnUrl=...basic_login_proxy..."}}}

4. GET  login.taobao.com/member/login_unusual.htm?user_en_id=...   （身份验证入口）
5. GET  passport.taobao.com/iv/mini/normal_validate.htm?firstIn=true&htoken=<htoken>
6. GET  passport.taobao.com/iv/mini/identity_verify.htm?htoken=<htoken>   （短信表单页）
7. GET  passport.taobao.com/iv/phone/send_code.do?type=phone&area=86&tag=86&phone=186****&htoken=<htoken>
        resp: {"content":{"code":0,"value":"验证码发送成功","success":true}}
8. POST passport.taobao.com/iv/mini/identity_verify.htm?htoken=<htoken>
        body: _tb_token_=... & action=verify_action & event_submit_do_validate=notNull
              & _fm.v._0.h=<htoken> & _fm.v._0.p=186**** & _fm.v._0.a=86
              & _fm.v._0.ph=<短信验证码> & _fm.v._0.pho=sms & bx-ua=234#...（IV 页另一个 bx-ua）
        → 验证通过 → trustLogin → basic_login_proxy → DChain 会话
```

IV（第 5–8 步）与项目现有 `protocol_login/iv_sms_solver.py:solve_iv_sms` 基本吻合，可复用。

## 二、纯协议复刻进展

- ✅ **端点**：用 `login.taobao.com`（非 havanalogin）的 login.jhtml 页 + 其 login.do。
- ✅ **密码编码**：`password2` 必须是 **RSA 密文的 HEX**（`cipher.hex()`），不是 base64。
- ✅ **check.do**：预检返回 `resultCode:100 processFinished:true`。
- ✅ 由此 **"请填写密码"（resultCode 100 软拒）已消除** —— 之前协议路打 havanalogin + base64 才会得到它。

## 三、当前卡点：x5sec / RGV587 设备指纹风控（硬墙）

`login.do` 返回 `RGV587_ERROR + x5secdata...punish`，**换代理 IP 无效**，确认是**设备指纹**问题，非 IP/账号标记。

根因：iv8 的 baxia 指纹不完整。真实浏览器登录时额外做了
- `umdcv4.taobao.com/repTw.json`、`repWd.json` —— umid **设备注册**（生成设备绑定 token）
- 持有 `cna`、`tfstk` 设备 Cookie

iv8 当前仅经 `ynuf.aliapp.org/service/um.json` 出 umidToken，`_cookies` 里**无 cna、无 tfstk**，
生成的 `bx-ua` 过不了 x5sec 校验。

### 突破方向（深度、未验证）
按 `iv8_补环境教程.md` §9：用 Proxy 拦截 `awsc_baxia.js` 在 iv8 中的执行，
查它读取/发起了哪些设备信号与网络请求，补齐 `umdcv4 repTw/repWd` 设备注册 + cna 获取，
使 iv8 产出 x5sec 认可的 `bx-ua`。x5sec 是阿里持续对抗的最硬反爬层，成功不保证。

### Phase 1 诊断结论（iv8 netLog 实测）
iv8 跑本地 baxia SDK 仅发起 4 个请求：`login.htm`、`et_f.js`、ynuf `wu.json`、ynuf `um.json`，
**完全没有 umdcv4/repTw/repWd**。补 `AWSC.use("um").init()` 也不触发 umdcv4。

线上 login.jhtml 的 viewData 关键配置：
- `umidServer = https://ynuf.alipay.com`
- `umidEncryptAppName = 090D1F110F187839282B151408`（umid 加密 appName）
- `nocaptchaAppKey = CF_APP_TBLogin_PC`
- 设备注册端点：`umdcv4.taobao.com/repTw.json`、`repWd.json`

**判定**：本地 `awsc_baxia.js/um.js`（2026-06-15）是旧的 ynuf 版，线上已用 umdcv4 设备注册。
本地 SDK 不会触发 umdcv4 → 设备未注册 → bx-ua 过不了 x5sec。

### Phase 2 进展（线上 SDK 实验）
抓到线上动态加载的整套 SDK（已下载到 `protocol_login/js_live/`）：
- `AWSC/AWSC/awsc.js`（10956B，动态加载器）
- `sd/baxia/2.5.36/baxiaCommon.js`（37374B）← 替代本地旧的 `awsc_baxia.js`（ynuf 版）
- `AWSC/et/1.84.2/et_f.js`、`AWSC/uab/1.140.0/collina.js`（与本地同版）
- `AWSC/WebUMID/1.93.0/um.js`（177654B，**与本地 um.js 同版** → um.js 不是问题）
- `AWSC/fireyejs/1.234.24/fireyejs.js`、`secdev/sufei_data/3.9.14/index.js`

**关键判定修正**：um.js/collina 与本地同版，差异在 **awsc 框架 + baxiaCommon 2.5.36**（本地是自带的 ynuf 版 awsc_baxia.js）。

实验（`debug_live_sdk_umdcv4.py` 思路）：iv8 里 `add_resource` 预注入整套模块 + 加载 awsc.js →
`AWSC` 对象创建成功、`AWSC.use("um")` 动态拉到 um.js（动态加载在 iv8 跑通）。
但 **`mod.init()` 回调返回 `timeout`** —— umid 采集在 iv8 内部超时未完成，故未发起 umdcv4。

### Phase 3 突破（umdcv4 在 iv8 跑通）★
关键发现：
1. **下载的 et_f.js / fireyejs.js / sufei index.js 是 gzip**（首字节 `1f8b08`），需先 gunzip。
   `js_live/` 里这三个已解压为明文 JS。
2. **不能依赖 `AWSC.use` 动态加载**（会 `use:timeout`，模块没注册成功）。
   正解：把整套 SDK **按依赖顺序直接 `ctx.eval`**（awsc → baxiaCommon → et_f → collina → um → fireyejs → sufei），
   同步注册后再 `AWSC.use("um").init({appName: 加密appName})`。
3. 这样 **um.init 返回 `cb:success`，并真实发起 `POST umdcv4.taobao.com/repTw.json`（设备注册）** 与
   `7i4gpa.tdum.alibaba.com/dss.js`（设备数据）。**umdcv4 在 iv8 里跑通了** —— x5sec 缺的那一环已打通。

复现脚本：`protocol_login/_wip_um_probe.py`（基座：`_wip_umdcv4_harness.py` + `js_live/`）。

### 当前精确卡点（Phase 4 起点）
`um.init` 成功但：
- `repTw.json` 重试 3 次后**没进 repWd**，设备注册未最终完成；
- `um.init` 成功回调 data 是 **`{}`**（这版 WebUMID 1.93.0 不从 init 回调返回 token）；
- 故 umidToken / bx-ua / cna 都还取不到。

根因方向：**repTw 的响应没被 um 消化**——iv8 的 `add_resource` 注入与 um 在 iv8 里读 POST XHR 响应的
机制没对上（按 URL 去重已排除，按序转发也无效）。需要：
1. 搞清 um 读 repTw 响应的方式（XHR? fetch? 自定义？），让注入的 `dt/ck` 被 um 正确接收 → 推进到 repWd → 完成设备注册。
2. 找 WebUMID 1.93.0 提取设备绑定 umidToken 的 API（非 init 回调）。
3. 取 bx-ua（configFY getUA，需设备流程完成后）+ cna（设备注册响应 Set-Cookie）。
4. 用设备注册后的 token/ua/cookie 打 login.do，验证 RGV587 消失、返回 iframeRedirectUrl。
5. 过 login.do 后接现有 `solve_iv_sms` → trustLogin → basic_login_proxy → DChain 会话。

## 四、现状结论

- **浏览器兜底路已稳定可用**：填充→短信（自动取码）→登录→写库，连续多次成功（见 `browser_fallback.py` + `slider_helper.py` + `sms_helper.py`）。
- **纯协议路**：协议结构已完全还原、端点/编码已攻克，仅剩 x5sec 设备指纹一关未过。
  若要纯协议，需投入 baxia 设备指纹复刻（多轮、深度逆向）。
